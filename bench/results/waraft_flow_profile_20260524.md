## Findings

- The real issue was WARaft generic command batches flushing independently while another generic flush was already in flight. This increased commit/apply pressure and kept connection processes waiting in `wa_raft_acceptor.call/3`.
- Making in-flight coalescing the default gives most of the measured win without adding a fixed batch window.
- After that fix, the remaining hot samples were mostly real commit waits plus Flow claim/apply work.
- A focused 5K-flow profile then showed Flow history projection rereading the same spilled WARaft apply-projection record once per generated value ref. Batching value hydration removes that stack.
- The next clean profile exposed `Flow.LMDB.history_index_key/3` using `String.pad/4` while publishing LMDB history locators. Replacing it with a fixed zero-prefix binary slice preserved key ordering and removed that sampled stack.
- Filtering generated value refs through the keydir avoids LMDB live checks for refs that were already projected and dematerialized. This removed a cold projection check from the sampled stack.
- The projector no longer waits on `LMDBWriter.flush/3`. Correctness is preserved because value projection writes the LMDB locator in the projector before deleting the keydir ref; waiting behind unrelated state/history LMDB work only made cold projection visible in the hot soak.
- History-index planning is now one pass. This addresses the sampled `Enum.map` work in `publish_history_index/3`, but the validation run did not show a clear workload-level win.
- For 5KB payloads, forcing the blob side-channel at 4KB is the wrong latency shape. It adds a blob append/fsync before the WARaft commit, which moved transition/claim calls into the 100-200ms range. With the production 256KB threshold, 5KB payloads stay inline and the same workflow completes with ~20-35ms command latency in this local run.
- The tradeoff is storage/memory: inline 5KB avoids the extra fsync and is much faster, but writes much more WARaft/Bitcask segment data. Blob side-channel still makes sense for genuinely large payloads where reducing replicated log bytes is worth the extra durable blob write.
- Current remaining hot samples are mostly WARaft storage workers and Flow history projectors doing real `lmdb_write_batch/3`/history-index projection work. The naive projector LMDB coalescing experiment did not help, so the next useful optimization target is either moving more projector planning/native LMDB work off BEAM schedulers or reducing the amount of history/value metadata that must be projected per transition.
- Flow hot-history eviction now uses a flat native delete NIF. The generic native `apply_batch` delete path was slower for this workload because most history keys are unique and Rustler had to decode nested `[{key, [member]}]` shapes.
- Flow history projection entries are stamped with the apply shard. That keeps the normal same-shard path from rehashing every generated history key before enqueueing projection.
- Native claim_due state-record apply now batches pending write/process-dictionary publication. The append order and rollback originals stay the same, but the hot path avoids repeated `Process.get/put` calls per claimed row.
- The stamped projection same-shard fast-path is intentionally narrow: it only applies to entries generated inside the current shard apply path. Unstamped entries still use the old route/grouping logic.
- Fast staged put apply now batches stale apply-projection cache deletion after successful ETS location updates. This removes a per-key `WARaftSegmentReader.delete_apply_projection_entries/3` call from the claim/transition apply path while preserving the old rollback boundary.
- Root-level apply-projection spill deletion looked attractive from the stack sample but was worse in the actual latency run. The current code stays with exact ref deletion for spill cleanup.
- Direct single-op native index dispatch also benchmarked worse. The generic `flow_index_apply_batch` path stays in place because it preserves ordering and performed better under the actual Flow workload.
- Projector hot-history/value keydir cleanup now reuses the row it already fetched for delete decisions. This removes a redundant ETS lookup in the sampled `track_keydir_binary_remove` path and improved the focused run.
- Smaller projector chunks are not automatically better. A 1024-entry chunk lowered residual keydir rows but increased projector/flush contention and worsened p99.
- A 4096-entry projector chunk is better on this workload: fewer LMDB/projection flush cycles, lower WARaft flush tail, and still bounded memory/keydir residue.
- The benchmark uses `MIX_ENV=dev --no-compile`, so changing a module default must be followed by a dev compile before a "default" run. Otherwise the app can silently benchmark stale beams while env-overridden runs look better.
- Flow history projection entries now carry direct value refs from apply. The projector still falls back to decoding encoded history records for recovered/old entries, but the hot path no longer reparses history fields just to dematerialize generated Flow values.
- The latest runs separate server latency from worker scheduling noise: server p99 stayed in the `<=40-60ms` class, while end-to-end workflow/sec varied mainly with empty claim volume. Runs with 50K+ claim calls and ~96% empty claims are measuring Python wake/claim behavior as much as FerricFlow apply latency.
- A tiny 1ms owner-wakeup coalesce window is enough to restore the focused workload to the best class without changing server code. The soak benchmark default now uses 1ms, because it turns many immediate low-yield claims into fewer fuller claims while only adding up to 1ms before the worker asks for work. The Python SDK default should be considered separately.
- The Python SDK should not inject client wall-clock `NOW` by default. Explicit `now_ms` is still supported, but omitted time should let the server/WARaft command clock decide. This fixes a semantic skew even though it was not by itself a throughput win in the focused benchmark.
- The raw all-states benchmark claim path had the same client `NOW` smell as the SDK. It now uses a tested helper that omits `NOW` by default, so `claim_states_mode=all` will measure server-clock behavior too.
- Router's native empty precheck must be conservative for unstamped claims. It now keeps explicit `NOW` exact, but for omitted `NOW` it uses a small 5ms lookahead only to decide whether it is safe to skip the write. That avoids dropping work in the precheck/apply timestamp gap; the actual claim still uses the state-machine apply timestamp.
- A wider 20ms precheck slack is not worth it for this workload. It increases near-future false positives and creates more empty client work.
